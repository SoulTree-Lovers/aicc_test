# 실패 유형별 개선 루프 운영 가이드

## 운영 목적
주간 리포트의 실패 패턴을 **데이터/프롬프트/정책/분류기** 네 축으로 빠르게 환류하여 품질 지표를 지속 개선한다.

## 공통 운영 사이클 (주간)
1. 실패 케이스 수집: Top 실패 패턴별 샘플 20~50건 추출.
2. 원인 라벨링: `retrieval`, `grounding`, `clarification`, `handoff`, `policy`.
3. 개선 액션 실행: 아래 유형별 플레이북 적용.
4. 재평가: 자동 지표 + 샘플 수동 검토.
5. 배포/롤백 판단: 게이트 기준 충족 여부 확인.

## 유형별 플레이북

### A. 데이터 보강
- 트리거: Retrieval miss 증가, 특정 intent recall 저하.
- 액션:
  - 누락 FAQ/정책 문서 색인 추가.
  - 동의어/오타/구어체 질의 증강.
  - hard negative 샘플 추가로 reranker 개선.
- 성공 조건:
  - 해당 intent의 근거정합성 +5%p 이상.

### B. 프롬프트 조정
- 트리거: 근거는 있으나 답변 포맷 오류/환각 발생.
- 액션:
  - 근거 인용 강제 규칙 강화.
  - 재질문 조건을 체크리스트로 명시.
  - 금지 응답 패턴(추측, 단정) 명문화.
- 성공 조건:
  - 정답성/근거정합성 동시 개선.

### C. 정책 수정
- 트리거: 정책 최신성 미반영, 부서 규정과 불일치.
- 액션:
  - 정책 변경 로그를 지식베이스 반영 SLA(예: 24h)로 운영.
  - 버전 필드(`policy_version`)를 응답 로그에 기록.
  - 정책 충돌 시 우선순위 테이블 업데이트.
- 성공 조건:
  - Policy staleness 실패 비중 50% 이상 감소.

### D. intent 분류기 개선
- 트리거: 오이관/미이관, 잘못된 intent 라우팅.
- 액션:
  - confusion matrix 상위 쌍 재학습.
  - 이관 필요 intent에 cost-sensitive loss 적용.
  - 저신뢰 구간 human-in-the-loop 라우팅.
- 성공 조건:
  - 이관 정확도 +5%p 이상, 치명 오이관 감소.

## 역할 및 책임
- Ops: 주간 리포트 발행, 액션 우선순위 결정.
- ML: 데이터/모델/프롬프트 실험 수행.
- Policy Owner: 정책 최신성 검수 및 승인.
- QA: 배포 전후 샘플 검증 및 회귀 체크.
